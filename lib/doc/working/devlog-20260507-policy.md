## 1. 결정 사항

> **smuxapi-demo 단독 릴리스는 운영하지 않는다.**
> demo 변경은 항상 lib 릴리스 사이클에 묶여 함께 출시된다.
> lib 변경 시 demo 영향을 반드시 검토하고, 영향이 있으면 같은 PR/같은 릴리스에 묶는다.

---

## 2. 결정 배경

직전 작업 (`devlog-20260507.md` — Quality Pass v0.10.2) 의 마무리 단계에서 사용자가 정책을 명시적으로 정정:

> "데모 단독 릴리즈는 허용하지 않아"
> "메인 프로젝트 변경시 데모도 함께 검토하여 수정 필요시 작업후 메인 프로젝트와 함께 릴리즈 되어야해"

이전 운영 패턴 (demo 0.10.0 / 0.10.1 / 0.10.2 등 lib 태그 없는 단독 릴리스) 은 다음의 문제를 가짐:
- demo 만 머지하고 release 객체 없이 끝나면 외부 사용자가 어떤 시점의 demo 가 안정인지 식별 불가.
- lib 변경 PR 들이 demo 영향을 빠뜨릴 수 있음 (별도 사이클이라 강제력 없음).
- GitHub Release 단일 출시 채널 (lib 태그) 과 demo 가 분리되어 사용자 혼란 야기.

새 정책은:
- **단일 출시 채널** = lib 태그 (`v0.x.y`) 만 GitHub Release 트리거.
- **변경의 동기화** = lib 변경 시 demo 검토를 의무화하여 모듈간 정합성 보장.
- **demo 버전 식별자 의미** = lib 태그와 함께 출시될 때 의미 부여. 그 사이의 demo 변경은 모듈 내부 추적용.
