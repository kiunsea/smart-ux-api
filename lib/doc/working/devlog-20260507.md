## 3. 작업별 상세

### 3.1 (A) `easy_kiosc_uif.json` + ChatRoomService graceful fallback

**원인 분석**:
- `ChatRoomService.getChatRoom()` 두 오버로드가 매번 새 ChatRoom 생성 시
  `ConfigLoader.loadConfigFromClasspath("easy_kiosc_uif.json")` 호출.
- 해당 자원이 demo main classpath 에 없어 `uifJson == null` →
  `"다음의 내용을 학습해 -> null"` 잘못된 학습 prompt 가 LLM 으로 전송됨.
- v0.9.6 의 Phase 3 IT (`FullScenarioTestCaseRealLlmIT`) 에서 의미있는 매칭 실패의 원인 중 하나.

**조치**:
- `lib/src/test/resources/test.easy_kiosc_uif.json` (UIF 스키마) 을 `smuxapi-demo/src/main/resources/easy_kiosc_uif.json` 으로 복사.
  - 학습 prompt 책임은 demo (UI 데모 도메인 지식) 이므로 lib 으로 옮기지 않음.
- 두 오버로드의 중복 try-catch 를 `sendUifLearningPrompt(ChatRoom)` 헬퍼로 추출.
  - `chatRoom == null` 가드
  - `uifJson == null` 시 `log.warn(...)` 후 학습 prompt 자체 생략 (graceful fallback)

### 3.2 (B) `BrowserLauncher` headless / no-browser 옵션

**원인 분석**:
- `BrowserLauncher.openBrowser()` Windows 분기에서 `Runtime.exec("rundll32 url.dll,FileProtocolHandler ...")` 실행.
- 비대화형 환경 (Windows headless / SSH session / `Start-Process` 로 띄운 child JVM) 에서 rundll32 spawn 후 부모 JVM 이 조기 종료되는 케이스 발견.
- Phase 3 IT / CI 에서 데모 자동 기동 불가능.

**조치**:
- `BrowserLauncher.launchWhenReady()` 의 무인자 오버로드는 빈 배열 위임 (하위 호환).
- 신규 `launchWhenReady(String[] args)` + 패키지-private `shouldSkipBrowser(String[] args)`.
- 4가지 분기 (OR):
  1. 환경변수 `SMUXAPI_NO_BROWSER=true`
  2. 실행 인자 `--smuxapi.no-browser` / `--smuxapi.no-browser=true`
  3. 시스템 프로퍼티 `java.awt.headless=true`
  4. `Desktop.isDesktopSupported()=false`
- `SmuxapiDemoApplication.main(args)` 의 호출을 `launchWhenReady(args)` 로 변경.
- 신규 `BrowserLauncherTest` 6건 — args / headless / null safety 분기 검증. (CI 환경 자체가 headless 일 수 있어 일부 단정은 약하게 처리.)

### 3.3 (C) GitHub Actions Node 20 deprecation 임시 우회

**원인 분석**:
- `actions/checkout@v4`, `actions/setup-java@v4`, `actions/upload-artifact@v4` 등이 Node 20 런타임 사용.
- 2026-09 이후 deprecated, 매 push 시 경고 노이즈 출력.

**조치**:
- 외부 actions 의 v5 를 일괄 적용하는 방식은 검증 부담이 커서 보류.
- 대신 핸드오프 권장 임시 처방 사용:
  - 3개 워크플로우 (`ci.yml`, `deploy-docs.yml`, `release.yml`) 에 워크플로우 레벨 env 추가:
    ```yaml
    env:
      FORCE_JAVASCRIPT_ACTIONS_TO_NODE24: 'true'
    ```
- 추후 각 action 의 v5 마이그레이션이 검증되면 별도 PR 에서 본 env 제거 + 버전 핀 갱신.

## 6. 본 세션에서 도입된 패턴 / 노하우

### Java 17 try-catch 헬퍼 추출 (NPE 가드 + null fallback)

`ChatRoomService.sendUifLearningPrompt(ChatRoom)` 처럼 동일 try-catch 가 두 오버로드에 중복될 때, 헬퍼로 추출하면서 동시에 두 가지 null 가드를 함께 추가하면 결함 방어 + 중복 제거가 동시에 일어남.

### Spring Boot CLI 옵션 자체 파싱

`String[] args` 를 main → Launcher 로 그대로 전달하면 Spring Boot 의 `--key=value` 컨벤션과 일관된 자체 옵션 (`--smuxapi.no-browser`) 을 가볍게 처리할 수 있음 (Spring 의 `Environment` 를 거치지 않고 startup 직전에 결정해야 하는 사이드이펙트 차단용).

### GitHub Actions Node 20 임시 처방

워크플로우 레벨 `env` 의 `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24` 는 모든 외부 action 을 일괄 강제 실행시키므로, v5 핀이 검증되지 않은 상황에서 가장 보수적인 noise reduction 수단.
